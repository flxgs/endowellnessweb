# Internationalization Workflow

This project uses `next-intl` with locale-based routing.

## Locales

- `en`
- `es`

## Where translations live

- `messages/en.json`
- `messages/es.json`

## Required rule for new UI strings

Every new user-facing string must be added as a translation key in **both** locale files.

Do not hardcode new UI text directly inside components.

## Add a new translation key

1. Add the key in `messages/en.json`.
2. Add the same key path in `messages/es.json`.
3. Add translated values for both locales.
4. Use the key in components via `next-intl` APIs (`useTranslations`, `getTranslations`, or message objects).

## Validation command

Run:

```bash
npm run i18n:check
```

This checks that both locale files have identical key paths.

`npm run build` already includes this check and fails if keys are missing in either locale.
